# Email Verification Process

## Overview

After a user registers, the app can require them to verify their email address before accessing protected routes. This is handled by Laravel Fortify combined with Laravel's built-in `MustVerifyEmail` interface.

**Note:** Email verification is currently configured but not enforced in this project. To activate it, the `User` model must implement `MustVerifyEmail` (see below).

---

## Enabling Verification

Two things must be true for verification to be enforced:

1. `Features::emailVerification()` is present in `config/fortify.php` — it already is.
2. The `User` model must implement the `MustVerifyEmail` interface. Currently this line is commented out in `app/Models/User.php`:

```php
// use Illuminate\Contracts\Auth\MustVerifyEmail;
```

Uncomment the import and add `implements MustVerifyEmail` to the class declaration to activate verification.

---

## The Verification Flow

### 1. User Registers

`app/Actions/Fortify/CreateNewUser.php` validates the input and creates the user record with `email_verified_at` set to `null` — unverified.

### 2. Verification Email Is Sent

Fortify fires a `Registered` event immediately after the user is created. Laravel listens for this event and, because the `User` model implements `MustVerifyEmail`, calls `$user->sendEmailVerificationNotification()`.

This sends an email containing a **signed URL** — a link that looks like:

```
https://yourapp.com/email/verify/42/a94a8fe5...?expires=1748476800&signature=8f3a2b...
```

The link contains three security components:

| Component | Purpose |
|---|---|
| `{id}` | Identifies which user to verify |
| `{hash}` | SHA-1 of the user's email — proves the link was issued for this address |
| `signature` | HMAC of the full URL signed with `APP_KEY` — prevents tampering |
| `expires` | Unix timestamp — the link is only valid for 60 minutes by default |

### 3. User Is Blocked Until Verified

Routes protected by the `verified` middleware (such as `/dashboard` and all `/settings` routes in `routes/web.php`) redirect any unverified user to the email verification notice page.

This page is rendered by `resources/js/Pages/Auth/VerifyEmail.vue` and tells the user to check their inbox.

### 4. Resending the Link

The verification page has a **Resend verification email** button. Clicking it POSTs to `route('verification.send')`. Fortify resends the email (rate-limited to once per 60 seconds) and flashes `verification-link-sent` to the session. The page then shows a green confirmation alert.

### 5. User Clicks the Link

Clicking the link in the email hits the `verification.verify` route:

```
GET  /email/verify/{id}/{hash}
```

Fortify's `VerifyEmailController`:
- Validates the URL signature (returns 403 if invalid or expired)
- Confirms the hash matches the user's current email address
- Sets `email_verified_at = now()` on the user record
- Redirects to `/dashboard`

## How the Email Is Actually Sent

Laravel does not send email itself — it composes the message and delegates delivery to a **transport**. The transport is configured via `MAIL_MAILER` in your `.env` file.

### Current Setup: `log` transport

The default in this project is:

```env
MAIL_MAILER=log
```

With the `log` transport no email is delivered. Instead, the full email content — including the verification URL — is written to `storage/logs/laravel.log`. This is the standard approach for local development: grab the link from the log to test verification without needing any email credentials.

### Transport Options

| Transport | How it works |
|---|---|
| `log` | Writes to `laravel.log`. Development only. |
| `array` | Stores in memory. Automated testing only. |
| `smtp` | Connects to any SMTP server over TCP. You manage delivery. |
| `sendmail` | Delegates to the server's local `sendmail` binary. |
| `mailgun` | API call to Mailgun's service. |
| `ses` / `ses-v2` | API call to AWS Simple Email Service. |
| `postmark` | API call to Postmark. |
| `resend` | API call to Resend. |

### Self-Managed vs Third-Party Services

**Self-managed (`smtp`, `sendmail`):** Your app connects directly to a mail server you control. You are responsible for DNS records (SPF, DKIM, DMARC), bounce handling, and deliverability reputation. Practical only for internal tools — difficult to do reliably for apps that send to public inboxes.

**Third-party services (`mailgun`, `ses`, `postmark`, `resend`):** Your app makes an HTTPS API call with credentials. Their infrastructure handles delivery to Gmail, Outlook, etc. They manage deliverability, bounces, and provide logs and dashboards. This is the standard approach for production apps.

### Recommended Services

- **Resend** — easiest setup, modern API, first-party Laravel support, 3,000 emails/month free.
- **Mailgun** — long-standing Laravel default, reliable, 100 emails/day free on the Flex plan.
- **Postmark** — best deliverability for transactional email (verification, password reset).
- **AWS SES** — cheapest at scale ($0.10 per 1,000 emails) but more complex to configure.

To switch to Resend in production, install the package and update `.env`:

```env
MAIL_MAILER=resend
RESEND_KEY=re_xxxxxxxxxxxx
MAIL_FROM_ADDRESS="noreply@yourdomain.com"
MAIL_FROM_NAME="Your App Name"
```
