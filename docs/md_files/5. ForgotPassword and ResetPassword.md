# Forgot Password Process

## Overview

This document covers both the Forgot Password and Reset Password flows end-to-end. The forgot password feature is handled entirely by Laravel Fortify. It is a two-stage process: the user first requests a reset link (via `ForgotPassword.vue`), and then uses that link to set a new password (via `ResetPassword.vue`). Fortify handles all backend logic — token generation, email delivery, token validation, and password update. The Vue pages are purely presentational.

The entire feature depends on `Features::resetPasswords()` being enabled in `config/fortify.php`. Without it, none of the related routes are registered.

---

## The `Features::resetPasswords()` Dependency

All four routes that make this feature work are wrapped in a single `if` block inside Fortify's route file (`vendor/laravel/fortify/routes/routes.php`):

```php
if (Features::enabled(Features::resetPasswords())) {
    Route::get('/forgot-password', ...)  ->name('password.request');
    Route::get('/reset-password/{token}', ...) ->name('password.reset');
    Route::post('/forgot-password', ...) ->name('password.email');
    Route::post('/reset-password', ...)  ->name('password.update');
}
```

If `Features::resetPasswords()` is removed from `config/fortify.php`, all four routes disappear. This means:

- `GET /forgot-password` returns a 404.
- `route('password.email')` in `ForgotPassword.vue` throws a Ziggy "route not found" error.
- The "Forgot your password?" link on the login page is hidden — `Login.vue` receives a `canResetPassword` prop that Fortify sets to `false` when the feature is disabled.

## Stage 1 — Requesting a Reset Link

### Step 1 — Rendering `/forgot-password`

When a guest visits `/forgot-password`, Fortify calls the closure registered in `app/Providers/FortifyServiceProvider.php`:

```php
Fortify::requestPasswordResetLinkView(fn () => Inertia::render('Auth/ForgotPassword', [
    'status' => session('status'),
]));
```

The `status` prop carries a flash message from the session. On first visit it is empty; after a successful submission it contains the "We have emailed your password reset link!" confirmation.

### Step 2 — The Frontend Form (`ForgotPassword.vue`)

The page renders a single email field inside `GuestLayout`. Inertia's `useForm` manages the field and tracks processing state:

```js
const form = useForm({ email: '' })
const submit = () => form.post(route('password.email'))
```

On submit, Inertia sends an XHR `POST` to the `password.email` route. The `PrimaryButton` reads `form.processing` to show a loading state while the request is in flight.

### Step 3 — Fortify Handles the POST

Fortify's `PasswordResetLinkController@store` receives the request and:

1. **Validates** that `email` is present and well-formed.
2. **Looks up** the user by email (lowercased, per `'lowercase_usernames' => true` in `config/fortify.php`).
3. **Generates a secure random token**, stores a bcrypt-hashed copy alongside the email and timestamp in the `password_reset_tokens` table.
4. **Emails the reset link** to the user. The link format is:
   ```
   /reset-password/{raw-token}?email=user@example.com
   ```
   The raw token travels in the URL; only its hash is stored in the database (similar to how passwords are handled).
5. **Flashes a `status` message** to the session and redirects back to `GET /forgot-password`.

**Note on user enumeration.** Whether or not the email belongs to a registered user, Fortify returns the same success-looking response. This prevents attackers from probing which addresses are registered.

The link is valid for 60 minutes by default (configurable in `config/auth.php` under `passwords.users.expire`).

### Summary Flow — Stage 1

```
GET /forgot-password
  → FortifyServiceProvider renders Auth/ForgotPassword via Inertia
  → status prop is empty on first visit

User enters email and submits

POST /forgot-password
  → PasswordResetLinkController validates email
  → Looks up user (email lowercased)
  → Generates token, stores hashed copy in password_reset_tokens
  → Sends reset link email to user
  → Flashes status message
  → Redirects back to GET /forgot-password

GET /forgot-password (again)
  → status prop now contains success message
  → Alert shown to user
```

## Stage 2 — Setting a New Password

### Step 1 — Rendering `/reset-password/{token}`

When the user clicks the link in the email, Fortify intercepts the request and runs the closure in `FortifyServiceProvider.php`:

```php
Fortify::resetPasswordView(fn (Request $request) => Inertia::render('Auth/ResetPassword', [
    'email' => $request->input('email'),
    'token' => $request->route('token'),
]));
```

Both the `email` (from the query string) and the `token` (from the URL segment) are passed as props so the Vue page can send them back on submit.

### Step 2 — The Frontend Form (`ResetPassword.vue`)

The email and token are held in the form state alongside the new password fields:

```js
const form = useForm({
    token: props.token,
    email: props.email,
    password: '',
    password_confirmation: '',
})

const submit = () => {
    form.post(route('password.update'), {
        onFinish: () => form.reset('password', 'password_confirmation'),
    })
}
```

The email field is visible (pre-filled) and editable. The token is not displayed but is submitted with the form. On completion, the password fields are cleared regardless of success or failure.

### Step 3 — Fortify Validates and Updates the Password

Fortify's `NewPasswordController@store` receives the request and:

1. **Validates** the token, email, password, and password_confirmation fields.
2. **Verifies the token** — looks up the `password_reset_tokens` row by email, then compares the submitted token against the stored hash. Also checks that the token has not expired.
3. **Updates the password** by delegating to the action bound in `FortifyServiceProvider.php`:
   ```php
   Fortify::resetUserPasswordsUsing(ResetUserPassword::class);
   ```
   `App\Actions\Fortify\ResetUserPassword` re-validates the password against `PasswordValidationRules` and writes `Hash::make($input['password'])` to the user record.
4. **Deletes the token** row from `password_reset_tokens` so the link cannot be reused.
5. **Redirects to `/login`** where the user can sign in with their new password.

### Failure Cases

| Cause | Outcome |
|---|---|
| Token expired | 422 with `errors.email = "This password reset token is invalid."` |
| Token already used | Same generic error |
| Email/token mismatch | Same generic error (no enumeration) |
| Password fails validation | 422 with `errors.password` |

Inertia routes the 422 response back into `form.errors`, which `<InputError>` renders under the relevant field.

### Summary Flow — Stage 2

```
User clicks link in email → GET /reset-password/{token}?email=...
  → FortifyServiceProvider renders Auth/ResetPassword via Inertia
  → email and token passed as props to the Vue page

User enters new password and submits

POST /reset-password
  → NewPasswordController validates token, email, password
  → Verifies token against hash in password_reset_tokens
  → Calls ResetUserPassword action → hashes and saves new password
  → Deletes token row (link is now single-use)
  → Redirects to /login
```

## Quick Reference

| Concern | File |
|---|---|
| Enable/disable the feature | `config/fortify.php` — `Features::resetPasswords()` |
| Register Fortify view closures | `app/Providers/FortifyServiceProvider.php` |
| Forgot-password page UI | `resources/js/Pages/Auth/ForgotPassword.vue` |
| Reset-password page UI | `resources/js/Pages/Auth/ResetPassword.vue` |
| Password update logic | `app/Actions/Fortify/ResetUserPassword.php` |
| Password complexity rules | `app/Actions/Fortify/PasswordValidationRules.php` |
| Token lifetime | `config/auth.php` → `passwords.users.expire` (default 60 min) |

## Named Routes

| Vue call | Verb + URL | Purpose |
|---|---|---|
| `route('password.request')` | `GET /forgot-password` | Render the request-link form |
| `route('password.email')` | `POST /forgot-password` | Submit email, trigger reset email |
| `route('password.reset')` | `GET /reset-password/{token}` | Render the new-password form |
| `route('password.update')` | `POST /reset-password` | Submit new password |
