# Navbar and Links

This guide explains how to add and configure the main navigation bar links in the
starter kit, and documents every option available for both the navbar as a whole
and individual links.

The whole system is driven by **one file** — `resources/js/navigation.js`. Add a
link there and it automatically appears in **both** the desktop bar and the mobile
hamburger dropdown. You never edit layout markup to add a link.

---

## The big picture

There are three pieces, but you normally only ever touch the first one:

| File | Role | Edit it? |
|------|------|----------|
| `resources/js/navigation.js` | Config: the link list + app-wide nav options | **Yes — this is your edit point** |
| `resources/js/Components/NavLinks.vue` | Renderer shared by desktop bar + mobile menu | Rarely |
| `resources/js/Layouts/AuthenticatedLayout.vue` | Places the bar, logo, hamburger, theme picker, avatar | Only to swap the logo |

Because a single component (`NavLinks.vue`) renders both the desktop bar and the
mobile dropdown from the same array, "add a link once → it shows in both places"
is always true. There is no second list to keep in sync.

---

## Quick start: add a link

Open `resources/js/navigation.js` and add a row to `navItems`:

```js
export const navItems = [
    { label: 'Dashboard', route: 'dashboard' },
    { label: 'Projects', route: 'projects.index' },   // <- new link
]
```

That's it. `Projects` now appears in the desktop bar and the mobile menu, points at
the `projects.index` Ziggy route, and lights up automatically when you're on that
route.

The only required fields are `label` and `route` (the Ziggy route name). Everything
else has a sensible default.

---

## Link options (the item shape)

Each entry in `navItems` is a plain object. Full shape:

```js
{
    label: 'Dashboard',        // REQUIRED: display text + accessible name
    route: 'dashboard',        // REQUIRED: Ziggy route name
    active: 'dashboard',       // active-match pattern (defaults to `route`)
    group: 'start',            // desktop segment: 'start' | 'center' | 'end'
    icon: LayoutDashboard,     // lucide component OR image path string
    iconPosition: 'left',      // 'left' (default) | 'right'
    iconOnly: false,           // if true, the icon replaces the label (desktop)
    tooltip: '',               // desktop hover bubble (appended to label if iconOnly)
    enabled: true,             // false hides it but keeps the row in the file
    show: (props) => true,     // runtime predicate over Inertia page props
    class: '',                 // escape-hatch utility classes
}
```

### Field reference

| Field | Required | Default | What it does |
|-------|----------|---------|--------------|
| `label` | **yes** | — | The text shown for the link, and its accessible name (the tooltip text too when `iconOnly`). Required on **every** link. A missing or whitespace-only label logs a dev-mode `console.warn` (no-op in production builds). |
| `route` | **yes** | — | The Ziggy route name the link navigates to. Used as the link target and, if `active` is omitted, as the active-match pattern. |
| `active` | no | the `route` value | The pattern passed to Ziggy's `route().current(...)` to decide if the link is "active". Lets you keep a link lit on routes other than its own target. **Supports wildcards**, e.g. `'reports.*'` keeps the link active on `reports.index`, `reports.show`, etc. |
| `group` | no | `'start'` | Which desktop segment the link sits in: `'start'`, `'center'`, or `'end'`. Ignored in the mobile dropdown (which is always a simple stacked list). |
| `icon` | no | — | An icon for the link. Either a **lucide component** or an **image path string** (png/gif/svg). See [Icons](#icons) below. |
| `iconPosition` | no | `'left'` | Which side of the label the icon sits on: `'left'` or `'right'`. |
| `iconOnly` | no | `false` | `true` renders only the icon on desktop (label hidden, shown as a tooltip + `aria-label`). The mobile menu always shows labels. |
| `tooltip` | no | — | A hover bubble shown on the **desktop bar only**. On a normal link it's the bubble text; on an `iconOnly` link it's appended to the label after an em dash (`Home — Go to your homepage`). The `aria-label` stays the bare label. See [Tooltips](#tooltips). |
| `enabled` | no | `true` | Set to `false` to hide the link without deleting the row. |
| `show` | no | — | A function `(props) => boolean`. Receives the Inertia page props. Return falsy to hide the link at runtime. This is the data-driven equivalent of `v-if` (auth/role/flag gating). |
| `class` | no | — | Extra utility classes appended to the rendered link — an escape hatch for rare custom spacing or styling. |

### `active` and wildcards

By default a link is active when you're on its own route. Use `active` when a link
should stay highlighted across a group of related routes:

```js
// "Reports" stays lit on reports.index, reports.show, reports.edit, ...
{ label: 'Reports', route: 'reports.index', active: 'reports.*' }
```

Under the hood this calls `route().current(item.active ?? item.route)`, so any
pattern Ziggy's `current()` accepts works here.

## Icons

A link can show an icon next to (or instead of) its label. There are **two ways**
to supply an icon, and the renderer auto-detects which you used by its type:

| You provide | Detected as | Rendered as |
|-------------|-------------|-------------|
| A **component** (e.g. a lucide icon) | not a string | `<component :is="icon">` |
| A **string** path (png / gif / svg file) | a string | `<img :src="icon">` |

### lucide icons (the kit's convention)

The starter kit already uses [`@lucide/vue`](https://lucide.dev) for all its icons.
Import the icon at the top of `navigation.js` and reference the component:

```js
import { LayoutDashboard, FileText } from '@lucide/vue'

export const navItems = [
    { label: 'Dashboard', route: 'dashboard', icon: LayoutDashboard },
    { label: 'Reports', route: 'reports.index', icon: FileText },
]
```

lucide icons inherit the current text colour, so they automatically match your
active styles and themes.

### Image files (png / gif / svg)

Pass a path string instead. Put the file under `public/` (served from the web root)
and reference it by URL:

```js
export const navItems = [
    { label: 'Docs', route: 'docs', icon: '/images/docs.svg' },
    { label: 'Status', route: 'status', icon: '/images/live.gif' },
]
```

Image icons are rendered at a fixed `h-5 w-5 object-contain` size.

### Position: left or right

`iconPosition` controls which side of the label the icon sits on. Default `'left'`:

```js
{ label: 'Next', route: 'next', icon: ChevronRight, iconPosition: 'right' }
```

### Icon-only links

Set `iconOnly: true` to drop the label and show just the icon on the desktop bar.
The `label` is **still required** — it becomes the link's tooltip and its
`aria-label`, so the link stays accessible and discoverable:

```js
{ label: 'Dashboard', route: 'dashboard', icon: Home, iconOnly: true }
```

> **Mobile note:** `iconOnly` only affects the **desktop** bar. The mobile dropdown
> is a vertical text menu, where icon-only rows would be unreadable, so it always
> shows the label (with the icon beside it, matching the avatar dropdown).

### Tooltips

Add a `tooltip` to give a link a daisyUI hover bubble. Tooltips render on the
**desktop bar only** — the mobile dropdown never shows them (touch has no hover,
and its labels are always visible).

```js
// Normal link: the bubble shows the tooltip text.
{ label: 'Dashboard', route: 'dashboard', tooltip: 'Your activity at a glance' }

// Icon-only, no tooltip: the bubble defaults to the label.
{ label: 'Home', route: 'home', icon: Home, iconOnly: true }

// Icon-only + tooltip: the tooltip is appended to the label after an em dash,
// so the bubble reads "Home — Go to your homepage".
{ label: 'Home', route: 'home', icon: Home, iconOnly: true, tooltip: 'Go to your homepage' }
```

How the bubble text (`data-tip`) is resolved on the desktop bar:

| Link | `tooltip` set? | Bubble shows |
|------|----------------|--------------|
| normal (labelled) | no  | *(no bubble)* |
| normal (labelled) | yes | the `tooltip` |
| `iconOnly` | no  | the `label` |
| `iconOnly` | yes | `label — tooltip` |

The **accessible name** screen readers announce (`aria-label`, only added when the
label is visually hidden) always stays the bare `label`. The appended tooltip text
is a visual affordance only, so the spoken name stays concise.

### Not supported: raw inline SVG strings

You cannot pass a raw `<svg>…</svg>` markup string — that would require `v-html`,
which is an XSS risk and against the kit's no-magic posture. Use an `.svg` **file**
(via the image-path form) or a lucide / Vue component instead.

## The three ways to hide or disable a link

There are three distinct situations, each with its own mechanism. Pick based on
*why* the link is hidden:

| Situation | Mechanism | Effect |
|-----------|-----------|--------|
| Not shipping yet / scratch link | Comment the row out (`//` or `/* */`) | Removed from the build entirely. |
| Exists but intentionally turned off | `enabled: false` | Skipped by the renderer, but the row stays visible in the file as a reminder. |
| Visibility depends on the user / role / flag at runtime | `show: (props) => …` | Rendered only when the predicate returns truthy. |

### Why there is no per-link `v-if`

The bar is rendered by a single `v-for` over `navItems`, so there is no individual
template line to attach a `v-if` to. The `show` predicate is the intended
equivalent, and it's actually better: the condition lives next to the link data
instead of in the layout markup.

```js
// Only show when a user is logged in
{ label: 'Account', route: 'account.edit', show: (props) => !!props.auth?.user }
```

The `props` argument is the Inertia page props object. In this starter kit the
shared auth shape is `props.auth.user` (with `id`, `name`, `email`), so auth gating
is `!!props.auth?.user`. Add your own shared props (roles, feature flags) in
`app/Http/Middleware/HandleInertiaRequests.php` and check them here.

> For a genuine one-off hand-written link, you can still drop a literal
> `<Link v-if="...">` directly into `AuthenticatedLayout.vue` outside the array.
> The array is for the links you want managed in one place.

## Navbar-wide options

These exports in `navigation.js` apply to the whole bar, not individual links.

### `activeStyle` — how the active link is highlighted

```js
export const activeStyle = 'daisy'   // 'daisy' | 'underline' | 'pill'
```

| Value | Look | Notes |
|-------|------|-------|
| `'daisy'` *(default)* | daisyUI's built-in active state — a soft filled block | Matches the Settings sidebar (`menu-active`), so the app feels consistent. |
| `'underline'` | A bottom border under the active link | The classic Laravel Breeze look. |
| `'pill'` | A rounded, saturated highlight (chip) | More prominent; good for marketing-style bars. |

The style applies app-wide and works in both the desktop bar and the mobile menu
(the renderer maps each style to appropriate classes for each context).

### `navJustify` — how the desktop link row is distributed

```js
export const navJustify = 'between'  // 'between' | 'start' | 'center' | 'end' | 'around'
```

This is the simple global lever for spreading the links across the bar. It maps to
flexbox `justify-content`:

| Value | Result |
|-------|--------|
| `'between'` *(default)* | Groups pushed to opposite ends, space between. |
| `'start'` | Links clustered on the left. |
| `'center'` | Links centered. |
| `'end'` | Links clustered on the right. |
| `'around'` | Equal space around each group. |

For finer control of individual links, combine `navJustify` with per-link `group`
(see below). `navJustify` only affects the desktop bar — the mobile dropdown is
always a simple stacked list.

### `navBreakpoint` — when links collapse to the hamburger

```js
export const navBreakpoint = 'lg'    // 'sm' | 'md' | 'lg' | 'xl'
```

At this Tailwind breakpoint and wider, the inline desktop links are shown. Below
it, the links collapse into the hamburger dropdown on the left of the bar.

> **Note:** only these four values are supported because the layout pre-declares
> the matching Tailwind classes (`hidden lg:flex`, `lg:hidden`, etc.). These class
> strings must be written out literally so Tailwind's scanner can generate them —
> a value built dynamically would be purged from the CSS. If you need a different
> breakpoint, add its literal class strings to the `breakpointClasses` map in
> `AuthenticatedLayout.vue`.

## Placing links with `group`

On desktop, each link is bucketed into one of three segments based on its `group`
value, rendered left-to-right inside a single flex row:

```
[ start group ........ center group ........ end group ]
```

Empty groups take no space, so the layout adapts to whatever you use. Combined with
`navJustify`, this covers most navbar shapes without custom CSS:

```js
export const navJustify = 'between'

export const navItems = [
    { label: 'Dashboard', route: 'dashboard', group: 'start' },
    { label: 'Projects',  route: 'projects.index', group: 'start' },
    { label: 'Pricing',   route: 'pricing', group: 'end' },
]
// Result: Dashboard Projects ............................ Pricing
```

If you only ever use `group: 'start'` (the default), you get a normal left-aligned
bar — you can ignore groups entirely until you need them. For anything groups +
`navJustify` can't express, the per-link `class` field is the escape hatch.

> `group` is ignored in the mobile dropdown — it always lists links top-to-bottom
> in array order.

## The logo

The **Flower** icon on the left of the bar is a **placeholder** brand/home link. It
is a normal Inertia `<Link>` pointing at `dashboard`, separate from the hamburger
button. To use your own logo, edit `AuthenticatedLayout.vue` — the relevant block
is marked with a comment:

```vue
<!-- Brand / home link. Placeholder logo — swap the Flower icon for your own. -->
<Link :href="route('dashboard')" class="btn btn-ghost btn-circle" aria-label="Dashboard">
    <Flower :size="24" />
</Link>
```

Swap the `<Flower />` icon for an `<img>`, an SVG, or a different lucide icon, and
change the `:href`/`aria-label` if your home route differs.

## Worked example

A bar with an auth-gated reports section (active across all its child routes), a
work-in-progress link that's turned off, and a right-aligned settings link:

```js
// resources/js/navigation.js
import { LayoutDashboard, FileText, Settings } from '@lucide/vue'

export const activeStyle = 'underline'
export const navJustify = 'between'
export const navBreakpoint = 'lg'

export const navItems = [
    { label: 'Dashboard', route: 'dashboard', group: 'start', icon: LayoutDashboard },

    // Lit on any reports.* route; only visible to logged-in users
    {
        label: 'Reports',
        route: 'reports.index',
        active: 'reports.*',
        group: 'start',
        show: (props) => !!props.auth?.user,
        icon: FileText,
    },

    // Built but not ready to ship — stays in the file, hidden from the bar
    { label: 'Billing', route: 'billing.index', group: 'start', enabled: false },

    // Icon-only, pushed to the right edge by navJustify: 'between'
    { label: 'Settings', route: 'profile.edit', group: 'end', icon: Settings, iconOnly: true },
]
```

## How it renders (at a glance)

```
Desktop (>= navBreakpoint):
  [Flower]  Dashboard  Reports ....................... [Theme] [Avatar v]

Mobile (< navBreakpoint):
  [hamburger]  [Flower] ............................. [Theme] [Avatar v]
        |
        tap -> dropdown lists the same links, vertically
```

The theme picker and avatar/logout dropdown on the right are part of the layout and
are unaffected by your `navItems` changes.

## Verifying your changes

1. Run `npm run dev` (or `composer dev` for the full stack).
2. Add or edit a link in `navItems`; it should appear in the desktop bar.
3. Navigate to the link's route — it should highlight per your `activeStyle`.
4. Shrink the browser below `navBreakpoint` — the hamburger appears and its
   dropdown lists the same links with the same active state.
5. Try `enabled: false` and a `show: () => false` link — both should disappear.
6. Give a link a wildcard `active` (e.g. `'reports.*'`) and visit a child route —
   the link should stay lit.
7. Add an `icon` (lucide component and/or an image path), try `iconPosition: 'right'`
   and `iconOnly: true` — the icon should render in both the bar and the dropdown,
   and the icon-only link should show its label as a tooltip on hover.
