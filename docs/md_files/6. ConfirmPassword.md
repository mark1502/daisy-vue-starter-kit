# 6. ConfirmPassword

## What is it?

ConfirmPassword is a "sudo mode" feature provided by Laravel Fortify. When a user tries to access a sensitive action (e.g. enabling two-factor authentication, deleting their account), Laravel checks whether they have recently re-entered their password. If not, they are redirected to a confirmation form before being allowed to proceed.

The confirmation is **not** a new login — it simply re-validates the current user's password against the database, then stamps the session with a timestamp. Future requests to protected routes check that timestamp and skip the prompt while it is still fresh.

---

## The Key Moving Parts

| Piece | Location | Role |
|---|---|---|
| `password.confirm` middleware | Laravel framework | Guards sensitive routes |
| `GET /user/confirm-password` | Fortify routes | Shows the confirmation form |
| `POST /user/confirm-password` | Fortify routes | Receives and validates the submitted password |
| `ConfirmablePasswordController` | `vendor/laravel/fortify` | Handles both routes |
| `ConfirmPassword` action | `vendor/laravel/fortify` | Runs the credential check |
| `ConfirmPassword.vue` | `resources/js/Pages/Auth/` | The Inertia page the user sees |
| `auth.password_timeout` | `config/auth.php` | How long (seconds) the confirmation stays valid (default: 10,800 / 3 hours) |

---

## Step-by-Step Workflow

### 1. User hits a protected route

A route is wrapped with the `password.confirm` middleware:

```php
Route::post('/user/two-factor-authentication', ...)
    ->middleware(['auth', 'password.confirm']);
```

Laravel's `RequirePassword` middleware reads `auth.password_confirmed_at` from the session. If the key is missing or the timestamp is older than `auth.password_timeout`, the request is intercepted and the user is redirected to `route('password.confirm')`. Laravel also saves the originally intended URL so it can redirect back to it after confirmation.

---

### 2. The confirmation form is displayed

**Route:** `GET /user/confirm-password` → `ConfirmablePasswordController@show`

The controller returns a `ConfirmPasswordViewResponse`. Because this project uses Inertia, that response renders the `Auth/ConfirmPassword` Inertia page component.

---

### 3. The Vue component

`resources/js/Pages/Auth/ConfirmPassword.vue` presents a single password input:

```js
const form = useForm({ password: '' })

const submit = () => {
    form.post(route('password.confirm'), {
        onFinish: () => form.reset(),
    })
}
```

`onFinish` clears the password field regardless of whether the attempt succeeded or failed — the input is never left populated after a submission.

---

### 4. The password is validated

**Route:** `POST /user/confirm-password` → `ConfirmablePasswordController@store`

The controller passes the submitted password to the `ConfirmPassword` action:

```php
$confirmed = app(ConfirmPassword::class)(
    $this->guard, $request->user(), $request->input('password')
);
```

Inside the action, unless a custom callback has been registered, it calls `$guard->validate()`:

```php
$guard->validate([
    $username => $user->{$username},  // e.g. 'email' => 'user@example.com'
    'password' => $password,
]);
```

This checks the credentials against the database **without** creating a new session or firing login events.

---

### 5a. Success

```php
if ($confirmed) {
    $request->session()->put('auth.password_confirmed_at', Date::now()->unix());
}
return app(PasswordConfirmedResponse::class);
```

The current Unix timestamp is written to the session. The response then calls:

```php
redirect()->intended(Fortify::redirects('password-confirmation'));
```

`redirect()->intended()` sends the user to the URL they originally tried to reach. If there was no intended URL (e.g. they navigated directly to the confirm page), they are sent to the configured fallback.

---

### 5b. Failure

```php
return back()->withErrors(['password' => 'The provided password was incorrect.']);
```

The user is sent back to the confirmation form. The error is displayed by `<InputError :message="form.errors.password" />` in the Vue component.

---

### 6. Subsequent requests — the timeout window

For every future request to a `password.confirm`-protected route, the middleware evaluates:

```
now() − session('auth.password_confirmed_at') < auth.password_timeout
```

While within the window the middleware passes through silently. Once the window expires the entire flow repeats from step 1.

## Customising the Behaviour

- **Timeout duration** — change `password_timeout` in `config/auth.php`.
- **Custom password check** — register a callback in `FortifyServiceProvider` via `Fortify::confirmPasswordsUsing(fn($user, $password) => ...)` to replace the default `guard->validate` logic (useful if passwords are hashed differently or checked via an external service).
- **Disable the prompt for a feature** — for two-factor auth routes, the `confirmPassword` option in `config/fortify.php` controls whether the `password.confirm` middleware is applied to those routes automatically.
