# Auth Routes and FortifyServiceProvider

## How Auth Routes Are Created

Fortify's auth routes are **not** defined in `routes/web.php`. Instead, they are registered automatically by Fortify itself when the application boots. The chain is:

### 1. `bootstrap/providers.php`

Laravel reads this file on every request and boots every listed service provider. Your app registers:

```php
App\Providers\FortifyServiceProvider::class,
```

This causes both Fortify's internal provider (`vendor/laravel/fortify/src/FortifyServiceProvider.php`) and your app's customization provider (`app/Providers/FortifyServiceProvider.php`) to run.

### 2. Fortify's Vendor Service Provider Loads the Routes

Fortify's own provider (in `vendor/`) calls `configureRoutes()` during boot:

```php
protected function configureRoutes()
{
    if (Fortify::$registersRoutes) {
        Route::group([...], function () {
            $this->loadRoutesFrom(__DIR__.'/../routes/routes.php');
        });
    }
}
```

This loads `vendor/laravel/fortify/routes/routes.php` — a route file inside the package that you never need to edit.

### 3. Routes Are Registered Conditionally Based on `config/fortify.php`

The route file checks which features are enabled before registering each group of routes. If a feature is not listed in your `config/fortify.php` features array, its routes simply don't exist.

## All Routes Fortify Registers

### Authentication (always registered)

| Method | URI | Name |
|--------|-----|------|
| GET | `/login` | `login` |
| POST | `/login` | `login.store` |
| POST | `/logout` | `logout` |

### Registration — requires `Features::registration()`

| Method | URI | Name |
|--------|-----|------|
| GET | `/register` | `register` |
| POST | `/register` | `register.store` |

```php
if (Features::enabled(Features::registration())) {
    Route::get('/register', [RegisteredUserController::class, 'create'])->name('register');
    Route::post('/register', [RegisteredUserController::class, 'store'])->name('register.store');
}
```

### Password Reset — requires `Features::resetPasswords()`

| Method | URI | Name |
|--------|-----|------|
| GET | `/forgot-password` | `password.request` |
| POST | `/forgot-password` | `password.email` |
| GET | `/reset-password/{token}` | `password.reset` |
| POST | `/reset-password` | `password.update` |

### Email Verification — requires `Features::emailVerification()`

| Method | URI | Name |
|--------|-----|------|
| GET | `/email/verify` | `verification.notice` |
| GET | `/email/verify/{id}/{hash}` | `verification.verify` |
| POST | `/email/verification-notification` | `verification.send` |

### Profile & Password Updates — requires respective features

| Method | URI | Name | Feature |
|--------|-----|------|---------|
| PUT | `/user/profile-information` | `user-profile-information.update` | `updateProfileInformation()` |
| PUT | `/user/password` | `user-password.update` | `updatePasswords()` |

### Password Confirmation (always registered)

| Method | URI | Name |
|--------|-----|------|
| GET | `/user/confirm-password` | `password.confirm` |
| POST | `/user/confirm-password` | `password.confirm.store` |
| GET | `/user/confirmed-password-status` | `password.confirmation` |

### Two-Factor Authentication — requires `Features::twoFactorAuthentication()` (disabled by default in this project)

Routes for challenge, enable, disable, QR code, secret key, and recovery codes. Currently commented out in `config/fortify.php`.

### Passkeys — requires `Features::passkeys()` (disabled by default in this project)

Routes for passkey login, confirmation, and management. Currently commented out in `config/fortify.php`.

All routes are wrapped in the `web` middleware group and guest routes are further protected by `guest:{guard}` middleware to prevent authenticated users accessing them.

## The Two FortifyServiceProviders

There are two service providers with similar names — it is important to understand which is which.

### `vendor/laravel/fortify/src/FortifyServiceProvider.php`

This ships with the Fortify package. You never edit it. Its responsibilities are:

- Loading the route file (as described above)
- Binding all of Fortify's response contracts to their default implementations in the service container
- Setting up the 2FA provider (Google2FA)
- Configuring passkeys integration

### `app/Providers/FortifyServiceProvider.php`

This is **your** app's customization layer, published into your codebase when Fortify was installed. It is listed in `bootstrap/providers.php` and runs on every request. Its responsibilities are:

**1. Binding your Action classes**

This tells Fortify which class to call for each auth operation:

```php
Fortify::createUsersUsing(CreateNewUser::class);
Fortify::updateUserProfileInformationUsing(UpdateUserProfileInformation::class);
Fortify::updateUserPasswordsUsing(UpdateUserPassword::class);
Fortify::resetUserPasswordsUsing(ResetUserPassword::class);
```

Without these bindings, Fortify would have no business logic to execute.

**2. Registering Inertia views**

Fortify is frontend-agnostic. It has no knowledge of Vue or Inertia. You explicitly tell it what to render for each auth page:

```php
Fortify::loginView(fn () => Inertia::render('Auth/Login', [
    'canResetPassword' => Features::enabled(Features::resetPasswords()),
    'status' => session('status'),
]));

Fortify::registerView(fn () => Inertia::render('Auth/Register'));
```

This replaces Fortify's default Blade view rendering with your Inertia responses.

**3. Configuring rate limiters**

Defines throttle rules for the login and 2FA endpoints:

```php
RateLimiter::for('login', function (Request $request) {
    $throttleKey = Str::transliterate(Str::lower($request->input(Fortify::username())).'|'.$request->ip());
    return Limit::perMinute(5)->by($throttleKey);
});
```

---

## Summary

Fortify's vendor provider handles the infrastructure — registering routes and binding contracts. Your app's `FortifyServiceProvider` handles customization — which classes execute the business logic, which Inertia pages to render, and how to throttle requests. The features enabled in `config/fortify.php` act as feature flags that control which routes and capabilities are available.
