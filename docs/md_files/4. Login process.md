# Login Process

This document walks through what happens when a user logs in to the starter kit, from the moment they hit `/login` to the moment they land on `/dashboard`. It assumes familiarity with Laravel and Vue but does not assume prior experience with Fortify or Inertia.

## The moving parts

- **Laravel Fortify** ‚Äî a headless auth backend. It registers all the auth routes (`/login`, `/logout`, `/register`, `/forgot-password`, etc.) and runs the credential-checking pipeline. It does **not** ship any UI.
- **Inertia.js v3** ‚Äî the glue between Laravel and Vue. Server-side, `Inertia::render('Auth/Login')` returns a JSON payload pointing at a Vue page; client-side, Inertia swaps Vue components without a full page reload.
- **Vue 3 + Tailwind 4 + daisyUI 5** ‚Äî the UI. The auth pages live in `resources/js/Pages/Auth/`.
- **Ziggy** ‚Äî exposes Laravel's named routes to JavaScript, so the Vue code can call `route('login')` instead of hard-coding `/login`.

Configuration highlights:

- `config/fortify.php:18` ‚Äî guard is `web` (standard session auth)
- `config/fortify.php:76` ‚Äî successful login redirects to `/dashboard`
- `config/fortify.php:164` ‚Äî registration, password reset, and email verification are enabled (2FA is not)

## Step 1 ‚Äî Rendering the login page

When the user visits `/login`, Fortify intercepts that route. Instead of returning a Blade view, it runs the closure registered in `app/Providers/FortifyServiceProvider.php:40`:

```php
Fortify::loginView(fn () => Inertia::render('Auth/Login', [
    'canResetPassword' => Features::enabled(Features::resetPasswords()),
    'status' => session('status'),
]));
```

Inertia returns the `Auth/Login` Vue page (`resources/js/Pages/Auth/Login.vue`) with two props. The page renders inside `GuestLayout` and uses daisyUI form controls.

## Step 2 ‚Äî Submitting the form

`Login.vue` uses Inertia's `useForm` composable to hold form state:

```js
const form = useForm({ email: '', password: '', remember: false })

const submit = () => {
    form.post(route('login'), {
        onFinish: () => form.reset('password'),
    })
}
```

When the user submits, `form.post(route('login'), ‚Ä¶)` sends an XHR `POST` to `/login` with:

- The form fields as JSON
- An `X-Inertia: true` header (tells Laravel to return an Inertia JSON response, not full HTML)
- The CSRF token (read automatically from the `XSRF-TOKEN` cookie)

While the request is in flight, `form.processing` is `true`, which `PrimaryButton` reads to show a loading state.

## Step 3 ‚Äî Fortify authenticates the request

The `POST /login` route hits Fortify's `AuthenticatedSessionController`, which runs two layers:

**Rate limiting.** Before validation, Fortify checks the `login` limiter defined in `FortifyServiceProvider.php:62` ‚Äî 5 attempts per minute per email+IP combination. Exceeding it returns a 429.

**Authentication pipeline.** A series of actions runs in order:

1. `EnsureLoginIsNotThrottled`
2. `RedirectIfTwoFactorAuthenticatable` ‚Äî no-op here since 2FA is disabled
3. `AttemptToAuthenticate` ‚Äî calls `Auth::guard('web')->attempt([...], $remember)`, which looks up the user by email (lowercased ‚Äî see `config/fortify.php:63`) and verifies the password against the bcrypt hash
4. `PrepareAuthenticatedSession` ‚Äî regenerates the session ID to prevent fixation, and (if `remember` was checked) sets a long-lived remember-me cookie

**On failure**, a `ValidationException` is thrown, Laravel returns 422 with `{ errors: { email: '‚Ä¶' } }`, Inertia populates `form.errors`, and the `<InputError>` component renders the message under the email field.

**On success**, Fortify returns a redirect to `/dashboard`.

## Step 4 ‚Äî Inertia turns the redirect into a SPA navigation

A normal Laravel 302 would cause the browser to do a full GET. Inertia changes that:

- After a non-GET request, Inertia upgrades the 302 to a **303 See Other** so the follow-up is guaranteed to be a GET (not a re-POST).
- The Inertia client intercepts the redirect and issues an XHR GET to `/dashboard` with `X-Inertia: true`.
- Laravel routes `/dashboard` through the `auth` + `verified` middleware (`routes/web.php:9`). The session cookie now identifies the just-authenticated user, so `auth` lets it through. `verified` checks `users.email_verified_at`; if the email isn't verified yet, the user is redirected to `Auth/VerifyEmail.vue`.
- The route closure returns `Inertia::render('Dashboard')`. Inertia's response middleware also runs `HandleInertiaRequests::share()` (`app/Http/Middleware/HandleInertiaRequests.php:36`), which attaches a global `auth.user` prop with `{ id, name, email }` so any Vue page can read the current user via `usePage().props.auth.user`.
- The browser receives JSON describing the new page, Inertia swaps the Vue component, and updates the URL via the History API ‚Äî no full page reload.

## Step 5 ‚Äî Staying logged in

Every subsequent request carries the encrypted session cookie. The `auth` middleware on `/dashboard` and the `/settings/*` group (`routes/web.php:9`) gates protected pages. Sensitive actions ‚Äî like changing the password or deleting the account (`routes/web.php:17`) ‚Äî additionally require re-entering the current password via Laravel's `current_password` rule, and Fortify can bounce the user to `Auth/ConfirmPassword.vue` when a "recently confirmed" window has expired.

Logout posts to the named `logout` route (also auto-registered by Fortify), which invalidates the session, rotates the CSRF token, and redirects to `/`.

| Wire Fortify views to Inertia pages, define rate limiters | `app/Providers/FortifyServiceProvider.php` |
